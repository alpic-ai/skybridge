---
title: requireBearerAuth
description: "Require a signed-in user on every request"
---

`requireBearerAuth` locks your server behind sign-in. Unauthenticated requests are turned away before any tool runs, and your tools receive the signed-in user.

## Example

Every tool requires a signed-in user with the `shop.read` scope: the middleware validates the token before any handler runs.

```ts server.ts highlight={5-8}
import { McpServer, requireBearerAuth } from "skybridge/server";

const server = new McpServer({ name: "shop", version: "1.0" }).use(
  "/mcp",
  requireBearerAuth({
    verifier: { verifyAccessToken },
    requiredScopes: ["shop.read"],
  }),
);
```

## Signature

```ts
requireBearerAuth(options: BearerAuthMiddlewareOptions): RequestHandler;
```

## Parameters

### `options`

```ts
type BearerAuthMiddlewareOptions = {
  verifier: OAuthTokenVerifier; // the token check you write, see Verifier below
  requiredScopes?: string[]; // every accepted token must carry all of these, or 403
  resourceMetadataUrl?: string; // absolute URL appended to WWW-Authenticate on 401, for discovery
};
```

- **`verifier`** the provider-specific token check you implement. See [Verifier](#verifier).
- **`requiredScopes`** a server-wide floor that layers under per-tool [`securitySchemes`](/api-reference/register-tool#securityschemes).
- **`resourceMetadataUrl`** points at your OAuth 2.0 [Protected Resource Metadata](https://datatracker.ietf.org/doc/html/rfc9728) so clients can discover the authorization server.

#### `verifier`

The `verifier` is the piece you write against your provider. Its only required method is `verifyAccessToken(token: string): Promise<AuthInfo>`:

- Resolve with an `AuthInfo` for a valid token. The middleware puts it on `extra.authInfo` for tool handlers.
- Throw `InvalidTokenError` to reject a malformed, badly signed, or expired token. The middleware returns a 401 with the right `WWW-Authenticate` header.

You do not check scopes here: the middleware enforces `requiredScopes` against `authInfo.scopes` and returns a 403 on a missing scope.

```ts
type AuthInfo = {
  token: string; // the raw bearer token
  clientId: string; // OAuth client_id (often the azp or client_id claim)
  scopes: string[]; // scopes granted; checked against requiredScopes and securitySchemes
  expiresAt: number; // unix seconds; required, requireBearerAuth rejects tokens with no expiration
  extra?: Record<string, unknown>; // anything else you want in handlers, e.g. sub, email
};
```

```ts
import { type AuthInfo, InvalidTokenError } from "skybridge/server";

async function verifyAccessToken(token: string): Promise<AuthInfo> {
  try {
    const payload = await validateToken(token); // JWT verification, introspection, etc.
    return {
      token,
      clientId: payload.client_id,
      scopes: payload.scope.split(" "),
      expiresAt: payload.exp,
      extra: { sub: payload.sub },
    };
  } catch (err) {
    throw new InvalidTokenError(
      err instanceof Error ? err.message : "Token validation failed",
    );
  }
}
```

<Info>
How you validate the token depends on your provider. Check its docs.
</Info>

## Returns

An Express `RequestHandler` to pass to [`server.use`](/api-reference/mcp-server#use), typically on `/mcp`.

- A valid token: the request proceeds, and handlers read it from `extra.authInfo`.
- A missing, invalid, or expired token: a 401 with a `WWW-Authenticate` header. A token missing a required scope: a 403.

<Info>
A valid token only proves who the caller is. Authorizing what they can do, and scoping data to them, stays the handler's job: read `extra.authInfo` and never trust a client-supplied identifier.
</Info>
